EXTERNAL DEPENDENCIES & THEIR ROLES:
════════════════════════════════════

1. NEWSAPI.ORG
   Role: News Source Database
   └─ What we use: fetch_news() function
   └─ API Endpoint: https://newsapi.org/v2/everything
   └─ Returns: Real articles from 10,000+ publishers
   └─ Rate Limit: 1,000 requests/day (free tier)

2. GOOGLE GEMINI API
   Role: AI Text Summarization
   └─ What we use: summarize_with_gemini() function
   └─ Model: gemini-pro
   └─ Capabilities: Reads text, generates summaries, understands context
   └─ Rate Limit: 60 requests/minute (free tier)

3. TEXTBLOB LIBRARY
   Role: Sentiment Analysis
   └─ What we use: get_sentiment() function
   └─ Local Processing: No API call needed (runs on your computer)
   └─ Method: Word polarity scoring
   └─ Speed: Milliseconds per article

4. CUSTOM CODE
   Role: Credibility Scoring
   └─ What we use: get_credibility_score() function
   └─ Method 1: Source reputation database (manually created)
   └─ Method 2: Language pattern analysis
   └─ Combined: Source score × (1 - sensationalism penalty)


PERFORMANCE CONSIDERATIONS:
═══════════════════════════

Sequential Processing Time:
- NewsAPI fetch: ~2 seconds (1 network request)
- Sentiment analysis: ~0.1 seconds per article × 10 = ~1 second
- Credibility scoring: ~0.01 seconds per article × 10 = ~0.1 second
- Gemini summaries: ~2 seconds per article × 10 = ~20 seconds (SLOW)
- Total with summaries: ~23 seconds
- Total without summaries: ~3 seconds

Caching Strategy:
- Results cached for 5 minutes
- Subsequent searches of same term return instantly
- User can force refresh with "Fetch & Analyze" button


ERROR HANDLING:
═══════════════

If NewsAPI fails:
→ Error message: "❌ Network error: [reason]"
→ App doesn't crash
→ User can try again

If Gemini API fails:
→ Summary shows: "Could not summarize: [error]"
→ Still shows description fallback
→ Rest of analysis continues

If .env keys missing:
→ NewsAPI error: "❌ NewsAPI key not found!"
→ Gemini disabled gracefully
→ App suggests creating .env file


SCALABILITY:
════════════

Current Architecture:
- Single user: Works fine
- Multiple users: Would need Load Balancing
- High traffic: Would need caching layer (Redis)
- Large dataset: Would need database (PostgreSQL)

For presentation: Single user is fine!
